# Naduri — 개발자용 컨텍스트 프라이머

> 출처: 내부 Google Docs `Naduri_dev_context_primer.md` (2026-07-22 통합)
> 사용법: 새 AI 대화/작업 맥락 앞에 이 규칙을 정렬시킨다. FE면 [공통]+[FE], BE면 [공통]+[BE].
> **아래 불변 규칙과 충돌하는 설계·코드 제안은 하지 않는다. 애매하면 규칙 편에서 보수적으로 판단한다.**

## [공통] 프로덕트 한 줄

Naduri = 걷기로 크립토 무경험자를 입장시켜, **걸어서 번 공돈으로 디파이를 처음 맛보게** 하고, 그 신뢰로 **원금 예치(→ 기와체인 TVL)** 까지 데려오는 모바일 앱. 기와체인(두나무/업비트의 EVM 호환 L2)과 업비트 정산 레일 위에서 돈다.

## [공통] 논리 사슬 (이걸 깨는 설계는 틀린 설계)

1. **걷기 = 리텐션 엔진** (금액이 아니라 매일 리셋되는 습관이 훅)
2. **디파이 = 회수 엔진** (진짜 매출은 rev share; 걷기 보상으론 못 번다)
3. **공돈 = 둘을 잇는 다리** (하우스 머니라 첫 예치 문턱이 0)
4. **광고 = 규모까지 버티는 연료** (걷기 보상 원가를 덮음)
5. **포인트 이연 = 런웨이** (테스트넷 동안 원금은 장부 숫자, 현금 유출은 이자만)

## [공통] 불변 규칙 (어겨선 안 되는 것)

1. **신뢰가 유일한 해자. 유저를 절대 속이지 않는다.** 가짜 잔고, 숨은 차감, 오해 유도 UI 금지.
2. **출금은 항상 원탭·진짜 열림.** 강요·마찰·"정말 나가실 건가요?"·인질형("출금하려면 입금") 금지.
3. **확정 수익률 약속 금지.** 미래 가치·이자는 항상 가정형 + "변동 가능" 표기. (규제: 유사수신·투자권유 회피)
4. **idle 잔고 데이터는 유저에게 되비추는 거울 용도만.** 서버 축적·광고주 판매 금지. 위치·의도 데이터 수집/판매도 안 한다.
5. **-EV 갓챠·사행성 뽑기 금지.**
6. **체인은 유저 눈앞에서 안 보이게.** 시드문구·가스·지갑·스왑을 UI에서 지운다. 유저는 "원화 넣었더니 이자"만 경험.
7. **인앱 크레딧은 폐쇄형(자사 전용) 유지.** 제3자 상품권·현금·코인 구매 수단으로 확장 금지. (선불전자지급수단·특금법 회피)
8. **토큰 발행은 기본이 아니다.** 요청 없이 토크노믹스·플랫폼 토큰을 제안하지 마라.

## [공통] 지금 단계 (테스트넷 / 파일럿 / gasok 2차)

**지금 만든다**: 걷기·적립, 오프체인 포인트 원장, 체험 예치 UX, 이자 시뮬레이션(우리 재원으로 지급), 출금 모멘트 전환 화면, 스마트컨트랙트 **"설계"**(문서/다이어그램).

**지금 안 만든다**: 실제 온체인 풀 연결, 업비트 계정 연동(파트너십 후), 실데이터 idle 개인화(연동 전엔 고정 예시), **광고 SDK**(파일럿에선 검증 오염되니 뺀다), 토큰.

**파일럿의 유일한 검증 지표**: 공돈 체험 → **첫 진짜 원금(₩1,000이라도) 전환율**. 설문 의향 아님, 실제 입금 행동으로.

## [FE] 프런트엔드

**스택 가정**: **React Native / Expo** (만보 센서 필요: `expo-sensors` Pedometer, iOS HealthKit / Android Google Fit 연동). 상태는 **로컬 우선**, 서버는 포인트 원장·이자 계산만. (팀 실제 스택에 맞게 조정)

**화면 흐름**: 온보딩 → 홈(걷기/적립, [받기]·[불리기] 상시) → 체험 예치 → 체험 대시보드 → ★출금 모멘트(전환) → 원금 전환. 상세는 [07-user-flow](./07-user-flow.md).

**FE가 반드시 지킬 것**:
- 온보딩·홈에서 크립토·지갑·퍼센트·코인명 노출 최소화. 이질감이 튀면 이탈.
- 출금 버튼은 **3탭 이내**, 확인 다이얼로그로 막지 말 것.
- **출금 모멘트 전환 화면이 사업의 심장**: 지금 출금 ₩X vs idle ₩Y를 뒀다면 10년 후 ₩Z (가정형·변동 가능) 두 줄. idle은 연동 전 고정 예시.
- 미래 수익률·이자는 **확정형 문구 금지, 항상 가정형**.
- 지갑 생성·가스·시드문구·스왑을 화면에 **절대 노출하지 않는다**(임베디드 월렛/AA 뒤로 숨김).

**요청하기 좋은 것**: 화면 컴포넌트, 상태 머신(적립→체험→전환), 만보 센서 연동, 이자 카운트업 애니메이션, 접근성(40~50 타겟이라 큰 글씨·명확한 대비).

## [BE] 백엔드

**핵심 구조**: **오프체인 포인트 원장이 1순위.** 걷기 적립·체험 예치·이자 발생·출금이 전부 원장 위 트랜잭션. 온체인은 파일럿에선 아직 아니다.

- **포인트 이연**: 걷기로 번 원금(포인트)은 테스트넷 동안 장부 숫자로만 쌓인다(현금 유출 X). 실제 지급은 메인넷에서 BTC/금으로. → 원장에 **"미지급 부채"**로 정확히 추적하고, 지급 재원(rev share 적립/토큰/두나무 지원 중 무엇인지)을 설계에 명시.
- **파일럿 이자**: 체험 예치 이자는 실제 디파이 수익률을 반영하되 **파일럿 재원(지원금)으로 지급**. 상시 원가가 아니라 **일회성 마케팅비로 회계 분리**. 메인넷에선 프로토콜이 이자를 발생시키고 우리는 rev share만 뗀다.
- **입금-이자-출금 루프**: 유저가 실제로 완주 가능해야 한다(출금 진짜 됨). 이 신뢰 경험이 전환의 조건.

**스마트컨트랙트 설계(gasok 제출용, EVM/Solidity 기준 — 기와 문서로 검증)**:
- 코어 루프를 컨트랙트로: 포인트/자산 예치 → 스테이블 풀 → 출금. 감사 가능·투명한 보상 풀.
- **풀 선정은 보수적으로.** 손실 회피 대중을 온체인 리스크에 중개하는 것 → 디페그·컨트랙트 리스크 한 번이 신뢰를 붕괴시킨다. **리스크 관리가 생명줄.**
- 업비트 정산·idle 연동은 파트너십 전제의 인터페이스로 **"설계만"** 해두고 배포는 미룬다.

**BE가 반드시 지킬 것**:
- idle 잔고 데이터: 조회용으로만, 서버 축적·판매 금지.
- 크레딧/포인트가 폐쇄형을 벗어나 제3자 결제·코인 구매로 새지 않게 **경계를 코드로 강제**.
- 크립토 지급(BTC/금)은 **업비트(라이선스 VASP)가 처리**하는 구조로. 우리가 직접 커스터디·환전하지 않는다.

**요청하기 좋은 것**: 포인트 원장 스키마·부채 추적, 이자 발생 배치, 출금 정산 흐름, Solidity 컨트랙트 스켈레톤(예치/보상/출금)과 보안 체크리스트, 파일럿 vs 메인넷 경계 설계.
